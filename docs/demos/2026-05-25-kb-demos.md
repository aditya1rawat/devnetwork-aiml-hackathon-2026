# Knowledge Base — Demo Scripts

**Goal:** show that Argus learns from past incidents. Three demos, each ~45-90s.
Lead with Demo 1 (the payoff), then Demo 2 (it compounds), close on Demo 3 (the visual).

---

## Prereqs (one-time, before any demo)

Bring up the full stack:

```bash
docker compose up -d neo4j     # graph store :7474 / :7687
pnpm dev:cluster               # mock cluster :7100-7104
pnpm dev:kb                    # KB :7300 (mcp) + :7301 (admin)
pnpm dev:orch                  # orchestrator :7200
pnpm dev:web                   # web :3000
```

Wait for two log lines before starting:
- KB: `Uvicorn running on http://0.0.0.0:7301`
- orchestrator: `[argus] incident-kb mcp connected`

Seed the KB (only when Gemini daily quota is fresh — takes ~15 min, rate-limited):

```bash
pnpm seed-kb                   # 12 historic incidents
```

Verify the graph is populated:

```bash
docker exec argus-neo4j cypher-shell -u neo4j -p devpass \
  "MATCH (n {group_id:'argus_incidents'}) RETURN count(n);"
```

> **Quota note:** agent *reasoning* runs on TFY Gateway (Claude + Nemotron), NOT
> Gemini. Only KB **ingest** (~10 Gemini calls each) and KB **search rerank**
> (small) touch Gemini. So the live demos below cost little Gemini quota — the
> expensive part is the one-time seed. Never seed live during a demo.

## Seed neighborhoods (why retrieval lands strong matches)

The two runnable live scenarios each sit in a dense cluster of seed history, so
the case graph renders rich rather than sparse.

**`worker-oom` (live) retrieves from:**
- `worker-oom-2024-q4-001` — heap leak, unflushed job buffer
- `worker-oom-2025-q1-006` — tracing buffer growth
- `worker-oom-2025-q1-012` — large payload batch (failed over)
- `memleak-2025-q1-009` — api response cache leak
- `cpu-saturation-2025-q1-008` — worker regex hot loop

**`db-saturation` (live) retrieves from:**
- `db-saturation-2024-q4-002` — N+1 query (failed over, sev1)
- `db-saturation-2025-q1-007` — missing index
- `network-partition-2024-q4-004` — worker ↔ db_proxy partition

Shared nodes that tie the graph together: `worker` / `db_proxy` service nodes,
`memleak` / `slow_query` root-cause nodes, `restart` / `scale` remediations.

## Demo 1 — "Agent consults history" (hero, ~90s)

**Setup:** KB pre-seeded. Web open at http://localhost:3000.

**Run:**
1. Click the **worker-oom** scenario tile.
2. Watch the **timeline**: within the first ~3 steps a `kb?` bar appears
   (agent invoked `read_incident_kb`) followed by `kb✓` (results returned).
3. Open the **reasoning pane** on that step — the returned prior cases appear in
   the step body. Point out the agent referencing a past worker-memory case.
4. Let it run to resolution.
5. **Final report** renders. Scroll to **"prior cases consulted"** — the case
   graph shows the current incident at center, 3-4 worker-memory neighbors, the
   shared `worker` service node, and a `memleak` root-cause node.
6. Click **⛶ fullscreen** for the judge shot.

**Talking point:** "Argus doesn't start cold. Mid-investigation it retrieves its
own past postmortems that share services, symptoms, or root causes — then
verifies against live signals before trusting them."

**Fallback:** if the agent doesn't call `read_incident_kb` this run (LLM
judgment), open a previously-resolved worker-oom incident from the incidents
list — its case graph is already rendered.

## Demo 2 — "The learning loop" (~60s)

**Setup:** KB seeded. Continues naturally from Demo 1.

**Run:**
1. Trigger **db-saturation**. Let it resolve.
2. Timeline shows a `kb+` bar at the end — `kb_ingest_queued`, the resolved
   incident being written back to the KB.
3. Immediately trigger **db-saturation** again.
4. In the second run's final report, the case graph now includes the incident
   you *just solved* as a neighbor node.

**Talking point:** "Every resolved incident makes the next investigation
smarter. Closed loop — resolve, ingest, retrieve — with no human curating a
knowledge base."

**Verify (optional, off-screen):** node count climbs after step 2:
```bash
docker exec argus-neo4j cypher-shell -u neo4j -p devpass \
  "MATCH (n {group_id:'argus_incidents'}) RETURN count(n);"
```

> Newly-ingested incident is searchable within ~30s of resolution. If running
> back-to-back fast, give ingest a moment before the second trigger.

## Demo 3 — "Cross-incident graph" (~45s, visual closer)

**Setup:** any resolved incident open.

**Run:**
1. Open the final report's case graph, click **⛶ fullscreen**.
2. Point at the `worker` (or `db_proxy`) **service node** fanning out to multiple
   incidents across time.
3. Trace the typed edges: incident →`involves`→ service, incident →`caused by`→
   root cause, incident →`remediated by`→ remediation.
4. Click a non-focus **incident node** — opens that incident in a new tab.

**Talking point:** "This isn't keyword search over text. It's a bi-temporal
knowledge graph — services, root causes, and fixes linked across every incident
Argus has ever closed."

## Optional — Cold vs Warm contrast

Strongest proof of value: same alert, two KB states, side by side.

- **Cold:** empty KB. Agent's `read_incident_kb` returns 0 hits, solves from live
  signals only, case graph = just the current node.
- **Warm:** seeded KB. Agent pulls 3-4 prior cases, richer reasoning, populated
  case graph.

**Do NOT seed live** (15-min rate-limited cost). Two safe stagings:

1. **Tab flip (recommended):** pre-stage two finished incident runs in separate
   browser tabs — one resolved while the KB was empty (sparse graph), one
   resolved while seeded (rich graph). Flip between them. Zero live seeding,
   zero quota risk.
2. **Recorded cold clip + live warm run.**

To produce the cold artifact ahead of time:
```bash
pnpm reset-kb                  # wipe
# trigger worker-oom in UI, let it resolve  -> sparse graph (tab A)
pnpm seed-kb                   # reseed (~15 min, offline)
# trigger worker-oom again, let it resolve  -> rich graph (tab B)
```

## Reset between rehearsals

- UI: **Reset KB** button in the chaos panel (confirm dialog).
- CLI: `pnpm reset-kb`

After a reset, re-seed before warm demos (`pnpm seed-kb`).

## Known limits (don't get caught on these)

- Only **worker-oom** and **db-saturation** run live. The other seed incidents
  (auth, config-drift, network-partition, cpu) exist as KB history only — great
  as graph neighbors, not triggerable. Live incidents always center on `worker`
  or `db_proxy`.
- To demo `auth`/`gateway` as a primary incident, a new chaos type +
  `DEMO_SCENARIOS` entry is needed (~30-45 min). Not required for the hero demos.
- Gemini free tier has a **daily** per-project request cap. Seeding all 12
  incidents (~120 calls) plus a few demo ingests fits a fresh day; if the cap is
  already spent, seeding fails until midnight Pacific. The token-bucket limiter
  + backoff handle the per-minute cap automatically.
